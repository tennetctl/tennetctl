# ADR-004: UUID v7 as Primary Keys

**Status:** Accepted
**Date:** 2026-03-29

---

## Context

tennetctl needs a primary key strategy for all tables. The options are:

- **Auto-increment integers** (`SERIAL`, `BIGSERIAL`): Simple, small, sequential
- **UUID v4**: Random 128-bit identifier, globally unique, widely supported
- **UUID v7**: Time-ordered 128-bit identifier, globally unique, monotonically increasing
- **ULID**: Similar goals to UUID v7 but a different encoding

The requirements are:
- Keys must be globally unique across all tennetctl instances (multi-tenant, potentially multiple deployments sharing the same schema)
- Keys should be safe to expose in URLs and API responses without leaking business intelligence (no sequential integers that reveal entity counts)
- Keys should perform well for Postgres B-tree indexing (sequential keys avoid index fragmentation)
- Keys should be usable in distributed SDK contexts where a client assigns the key before inserting

---

## Decision

All primary keys use UUID v7, generated via the `uuid-utils` Python library (`uuid_utils.uuid7()`).

UUID v7 is a 128-bit identifier where the first 48 bits are a Unix timestamp in milliseconds. This means:
- Keys are time-ordered (monotonically increasing within the same millisecond)
- New rows are inserted at the end of the B-tree index, not randomly distributed
- The creation time of any entity is embedded in its ID and can be extracted without a database lookup

The Postgres column type is `UUID`. Keys are generated in Python before insertion (not in the database).

---

## Consequences

**Positive:**
- No index fragmentation from random key insertion (unlike UUID v4)
- Approximate creation time is always available from the ID alone
- Keys are safe in URLs — no sequential enumeration possible
- Works across distributed insertions (SDK-generated keys, batch imports) without coordination
- UUID is natively indexed and stored efficiently in Postgres (16 bytes, not text)

**Negative:**
- UUID v7 is newer (RFC 9562, 2024) — not all tooling supports it natively yet
- Keys generated within the same millisecond have random ordering within that millisecond (not a problem in practice)
- Slightly more complex to generate than `gen_random_uuid()` (requires `uuid-utils` library)

**Mitigations:**
- UUID v7 generation is centralised in `01_core/id.py` — `from core.id import new_id`. One import, used everywhere.
- UUID v7 is backward-compatible with UUID v4 in all Postgres contexts — any tooling that accepts UUID accepts UUID v7
- The timestamp embedded in UUID v7 is not a substitute for a proper `created_at` column — both are present on every table

---

## Alternatives Considered

**Auto-increment integers (BIGSERIAL):** Rejected. Sequential integers in URLs and API responses leak entity counts (users can tell how many customers you have by looking at their user ID). Not safe to expose in external APIs.

**UUID v4:** Rejected. Random UUIDs cause B-tree index fragmentation — every new row inserts into a random position in the index, causing frequent page splits. At tennetctl scale (millions of rows per table) this becomes a performance problem.

**ULID:** Similar properties to UUID v7 (time-ordered, unique, safe to expose) but a different encoding (Crockford Base32 string rather than UUID hex format). Rejected because UUID is the standard Postgres type with native support, while ULID requires storing as text or with a custom type.

**nanoid / custom short IDs:** Rejected. The collision resistance of 128-bit IDs is well-understood and proven. Shorter IDs have higher collision probability and are not worth the trade-off.

---

## Usage

```python
# 01_core/id.py
import uuid_utils

def new_id() -> uuid_utils.UUID:
    """Generate a new UUID v7 primary key."""
    return uuid_utils.uuid7()

# In repository functions:
from core.id import new_id

user_id = new_id()
await conn.execute(
    'INSERT INTO "02_iam".fct_users (id, email) VALUES ($1, $2)',
    user_id, email
)
```

The key is generated in Python before the INSERT, not with `gen_random_uuid()` in SQL. This allows the caller to know the ID before the database confirms the insert, which is important for building the response and for distributed tracing.
