## Planned: JWT RS256 + JWKS Endpoint

**Severity if unbuilt:** MEDIUM (HS256 works but prevents external token verification)
**Depends on:** vault (key storage), authlib library

## Problem

JWT signing uses HS256 (symmetric secret). Any service that needs to verify
a token must share the secret — not viable for microservices or third-party
integrations. RS256 allows public-key verification without exposing the
signing key.

## Scope when built

### Migration path (three steps, each independently deployable)

1. **Swap HS256 to authlib**: Replace homebrew `jwt_utils.py` with
   `authlib.jose`. Behaviour identical, interface stable (`issue_token` /
   `verify_token`). No route changes.

2. **Add RS256 key pair**: Generate RSA-2048 key pair at vault init.
   Store private key at `tennetctl/iam/jwt_private_key` (encrypted).
   Store public key at `tennetctl/iam/jwt_public_key` (plaintext is fine).
   `jwt_utils.py` reads private key for signing, public key for verification.

3. **JWKS endpoint**: `GET /.well-known/jwks.json` returns the public key
   in JWK format. No auth required. Used by external services to verify
   tokens without calling back to tennetctl.

### Key rotation

- Vault stores key pairs with a version label (e.g. `v1`, `v2`).
- JWT `kid` header carries the version.
- `verify_token` tries current key, then falls back to previous key during
  overlap window (default 24h).
- JWKS endpoint returns both current and previous public keys.
- Old private key can be deleted after overlap window.
- See `02_vault/06_planned/01_key_rotation.md` for vault-side rotation plan.

## Not in scope here

- OIDC discovery endpoint (`/.well-known/openid-configuration`) — part of
  OAuth2/OIDC provider sub-feature
- Social login — depends on authlib OAuth2 client (separate sprint)
