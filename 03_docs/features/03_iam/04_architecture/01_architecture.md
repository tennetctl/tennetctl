# IAM — Architecture

> **V1 scope:** minimum IAM for a single admin to log in and use the
> product. Full IAM (user management, RBAC, MFA, orgs) is a later module.
> Deferred items live in `06_planned/`.

---

## Schema layout

All tables live in the `"03_iam"` Postgres schema. No cross-schema foreign
keys (R-002 — cross-module data flows via events).

```text
03_iam
├── 06_dim_entity_types         iam_user | iam_session
├── 06_dim_account_types        default_admin | default_user (stub)
├── 07_dim_attr_defs            username | email | password_hash |
│                               token_hash | ip_address | user_agent |
│                               refresh (JTI) | jti (access JTI)
├── 07_dim_auth_types           username_password
├── 07_dim_token_types          access | refresh
├── 08_dim_session_statuses     active | revoked | expired
├── 10_fct_users                User identity — id, org_id, account_type_id, auth_type_id
├── 20_fct_sessions             Session — id, user_id, status_id,
│                               token_prefix, refresh_token_hash,
│                               refresh_token_prefix, refresh_expires_at,
│                               expires_at, absolute_expires_at
└── 20_dtl_attrs                EAV attributes (username, email, password_hash,
                                token_hash, ip_address, user_agent, jti, refresh JTI)
```

Views:

```text
v_users     → users with username + email resolved from EAV
v_sessions  → sessions with status + prefix columns resolved
```

## Token model

V1 uses a **two-token scheme** for the single admin:

| Token | Type | TTL | Storage | Verified by |
| ----- | ---- | --- | ------- | ----------- |
| Access token | JWT (HS256) | 15 min | Client only (memory / `Authorization` header) | Stateless signature check |
| Refresh token | Opaque random | 7 days | HttpOnly cookie `tcc_refresh` | DB lookup: prefix index → Argon2id verify |

### Why JWT for access + opaque for refresh

- **Access token short-lived → stateless verification is fine.** No DB
  hit on every authenticated API call. The session middleware validates
  the JWT signature + `exp` claim without touching Postgres.
- **Refresh token long-lived → must be revocable.** Opaque tokens stored
  as Argon2id hashes let us hard-revoke on logout with no token blacklist.
  JWTs cannot be revoked without a blacklist, making a 7-day JWT dangerous.

### JWT signing key

- Stored in Vault (`02_vault`) under the path `iam/jwt_signing_key`.
- The IAM backend loads it at startup via the Vault client. If Vault is
  sealed, the backend refuses to start (same as it refuses DB connections).
- Key type: 256-bit random bytes (generated by `00_setup/02_vault_init`
  `--generate-keys` flag). See `00_setup/05_sub_features/02_vault_init/`.
- HMAC-SHA256 (HS256). No RS256 in v1 — there are no third-party consumers
  of the token, so asymmetric keys add complexity with no benefit.
- Key rotation: manual for v1. Rotate by generating a new key in Vault and
  restarting the backend. All existing access tokens (15 min TTL) expire
  naturally; active users get a new token on their next refresh.

### JWT claims

```json
{
  "sub":  "<user_id UUID v7>",
  "sid":  "<session_id UUID v7>",
  "jti":  "<random UUID v7 — unique per token issuance>",
  "iat":  <issued-at Unix timestamp>,
  "exp":  <iat + 900 (15 minutes)>,
  "nbf":  <iat>
}
```

`sub` = user identity. `sid` = session identity (enables session revocation
check if needed). `jti` stored as an EAV attr on the session for audit
correlation; not used for revocation in v1 (access tokens expire in 15 min).

## Authentication flow

### Login

```text
POST /v1/auth/login  { username, password }
    │
    ▼
1. Repository: resolve user by username
   SELECT id, username, email, account_type
     FROM "03_iam".v_users
    WHERE username = $1
      AND is_deleted = FALSE AND is_active = TRUE
    │
    ▼
2. Timing-safe compare: verify_password(stored_hash, plaintext)
   If user not found: run verify_password against dummy hash to equalise
   latency, then raise InvalidCredentials (no username enumeration).
    │
    ├── failure → emit iam.login.failed, return 401 INVALID_CREDENTIALS
    │
    ▼
3. Generate tokens
   access_raw  = build and sign JWT (claims above, signed with Vault key)
   refresh_raw = secrets.token_urlsafe(48)   # 384-bit opaque token
    │
    ▼
4. Persist session
   INSERT INTO 20_fct_sessions (
       id, user_id, status_id,
       token_prefix,           -- first 16 chars of access_raw (for future index use)
       refresh_token_hash,     -- argon2id(refresh_raw)
       refresh_token_prefix,   -- first 16 chars of refresh_raw
       refresh_expires_at,     -- now() + 7 days
       expires_at,             -- now() + 7 days (mirrors refresh; slides on use)
       absolute_expires_at,    -- now() + 30 days
       last_seen_at, created_by, updated_by
   )
   Insert EAV attrs: token_hash=argon2id(access_raw), ip_address, user_agent, jti
    │
    ▼
5. Set cookies (HttpOnly, SameSite=Strict, Path=/)
   tcc_refresh=<refresh_raw>; Max-Age=604800; Secure (prod only)
   (Access token is NOT set as a cookie — returned in response body only)
    │
    ▼
6. Return 200
   { ok: true, data: { access_token: "<jwt>", token_type: "bearer",
                        user: { id, username, email } } }
   Raw tokens leave memory after this response. Only hashes remain in DB.
    │
    ▼
   Emit iam.user.logged_in
```

### Request authentication (session middleware)

Every non-exempt request carries the JWT in `Authorization: Bearer <token>`.

```text
Request with Authorization: Bearer <jwt>
    │
    ▼
1. Verify JWT signature with Vault signing key.
   On failure → 401 INVALID_TOKEN
    │
    ▼
2. Check exp, nbf, iat claims.
   If exp < now() → 401 TOKEN_EXPIRED (client should refresh)
    │
    ▼
3. Extract sub (user_id) and sid (session_id).
   Optionally fetch session row to check deleted_at / status = active.
   V1 does a lightweight SELECT on session id (single PK lookup, not O(N)).
    │
    ▼
4. Inject user_id + session_id into request.state.
5. Call next handler.
```

Exempt paths (no auth required):

```text
/healthz
/v1/auth/login
/v1/auth/refresh
/v1/vault/status
/v1/vault/unseal
/v1/vault/seal
```

### Token refresh

```text
POST /v1/auth/refresh   (no body; tcc_refresh cookie required)
    │
    ▼
1. Read tcc_refresh cookie.
   If missing → 401 REFRESH_TOKEN_REQUIRED
    │
    ▼
2. Find candidate sessions:
   SELECT id, user_id, refresh_token_hash, refresh_expires_at,
          absolute_expires_at
     FROM "03_iam"."20_fct_sessions"
    WHERE refresh_token_prefix = <first 16 chars of cookie>
      AND deleted_at IS NULL
      AND status_id = :active
   (prefix index makes this O(log N), typically 1 row)
    │
    ▼
3. Argon2id verify: verify_password(candidate.refresh_token_hash, cookie_value)
   No match → 401 INVALID_REFRESH_TOKEN
    │
    ▼
4. Check expiry:
   IF now() > refresh_expires_at   → 401 REFRESH_TOKEN_EXPIRED
   IF now() > absolute_expires_at  → 401 SESSION_EXPIRED (must re-login)
    │
    ▼
5. Rotate refresh token:
   new_refresh_raw    = secrets.token_urlsafe(48)
   new_refresh_hash   = argon2id(new_refresh_raw)
   new_refresh_prefix = new_refresh_raw[:16]
   UPDATE 20_fct_sessions SET
       refresh_token_hash   = new_refresh_hash,
       refresh_token_prefix = new_refresh_prefix,
       last_seen_at         = now(),
       updated_at           = now()
    WHERE id = <session_id>
    │
    ▼
6. Issue new access token (new jti, new exp = now() + 15 min).
   Update EAV jti attr for the session.
    │
    ▼
7. Set-Cookie: tcc_refresh=<new_refresh_raw>; HttpOnly; Secure; SameSite=Strict
8. Return 200:
   { ok: true, data: { access_token: "<new_jwt>", token_type: "bearer" } }
    │
    ▼
   (No audit event for routine refresh — too noisy. Future: emit on anomaly.)
```

The old refresh token is invalidated the moment the UPDATE commits. Any
replay of the old token fails step 3 (hash no longer matches). Refresh token
family tracking + compromise detection is deferred to `06_planned/07_refresh_family_id.md`.

### Logout

```text
POST /v1/auth/logout   (Authorization: Bearer <jwt> required)
    │
    ▼
1. Session middleware validates JWT → session_id in request.state
    │
    ▼
2. UPDATE 20_fct_sessions
      SET status_id  = :revoked,
          deleted_at = now(),
          updated_at = now()
    WHERE id = <session_id>
    │
    ▼
3. Clear cookies:
   Set-Cookie: tcc_refresh=; Max-Age=0; Path=/
4. Emit iam.user.logged_out
5. Return 204
```

## Password hashing

Argon2id parameters and `hash_password` / `verify_password` live in
`backend/01_core/password.py`. Both `03_iam.01_auth` (login, refresh token
hashing) and `00_setup.03_first_admin` (first admin password) import from
here.

Same parameters are used for user passwords and refresh token hashes — a
compromised DB gives an attacker no shortcut on tokens vs. passwords.

PHC-format output encodes parameters per row so future parameter upgrades
don't invalidate existing hashes. Lazy re-hash on login is planned
(`06_planned/08_argon2_rehash.md`).

## Key invariants

1. `10_fct_users.id` is UUID v7. `org_id = id` for the first admin
   (reflexive singleton org).
2. Username, email, and password_hash are EAV attributes. No strings on
   `10_fct_users`.
3. Access token is never persisted. Its hash lives in EAV (`token_hash`
   attr) for audit only, not for verification (verification is stateless
   via JWT signature).
4. Refresh token raw value is never persisted. Its Argon2id hash lives on
   the `20_fct_sessions` row. The raw value exists in memory only for the
   duration of the login or refresh response.
5. `token_prefix` and `refresh_token_prefix` are the first 16 characters
   of the respective raw tokens. Not secret. Used as index predicates only.
6. `refresh_expires_at` (7d) is hard. `absolute_expires_at` (30d) is
   harder. The stricter one governs when re-login is required.
7. Logout hard-revokes the session row immediately. Any in-flight access
   tokens expire within 15 minutes naturally.
8. JWT signing key comes from Vault. If Vault is sealed, the backend will
   not issue or verify tokens.

## Security properties

| Threat | Mitigation |
| ------ | ---------- |
| Password brute force | Argon2id 64MB/3-iter; rate limiting planned (`06_planned/01_rate_limiting.md`) |
| Refresh token theft (DB read) | Argon2id hash — raw token not recoverable |
| Access token theft (XSS) | Token in response body, not cookie — frontend must store in memory only, not localStorage |
| Refresh token theft (XSS) | HttpOnly cookie — JS cannot read it |
| Refresh token theft (MITM) | Secure flag — HTTPS only in prod |
| CSRF on refresh | SameSite=Strict — cookie not sent cross-origin |
| Username enumeration | Identical error + latency for "not found" vs "wrong password" |
| JWT algorithm confusion | HS256 only, key loaded from Vault — no `alg: none`, no RS256 downgrade |
| JWT replay after logout | 15-min TTL means at most 15 min of post-logout access; full revocation via family ID is planned |
| Session fixation | New session row on every login |
| Indefinite session | absolute_expires_at hard cap at 30d |
| Compromised refresh token | Rotation on every use; family invalidation planned (`06_planned/07_refresh_family_id.md`) |
| DB stolen, offline crack | Argon2id 64MB × 3 iterations on both passwords and refresh tokens |
