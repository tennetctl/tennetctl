## First Admin — Scope

## What it does

Phase 3 of the install wizard. Creates the first administrative user in
`03_iam.10_fct_users`, hashes their password with Argon2id, and in the
same transaction flips `system_meta.installed_at` to the current
timestamp. Every subsequent feature's boot-time check is
`WHERE installed_at IS NOT NULL`, and that column is written here —
but the install is not *complete* until Phase 4 seeds the mandatory
runtime rows into `10_fct_settings`.

After Phase 3 commits, the operator has exactly one login that will
work once Phase 4 finishes and the server starts.

## Prompts

```text
First admin username    (3-64 chars, letters/digits/underscore/dot/hyphen):
First admin email       (valid email address, used for future password resets):
First admin password    (12+ chars, at least one digit or symbol):
Confirm password        (must match):
```

Validation:

- **username**: `^[A-Za-z0-9._-]{3,64}$` — ASCII only, no whitespace
- **email**: basic RFC 5322-lite check via Pydantic `EmailStr`; the
  wizard does not send a verification email at install time (that is a
  future enhancement)
- **password**: minimum length 12, must contain at least one non-alpha
  character. The wizard does **not** impose a maximum length — Argon2id
  handles arbitrarily long inputs and length caps just annoy operators
  using a password manager
- password and confirmation must match byte-for-byte

Non-interactive mode (`--admin-username`, `--admin-email`,
`--admin-password`) bypasses the prompts but runs the same validators.
The wizard refuses to run non-interactively with a short or missing
password — there is no `--force-weak-password` escape hatch.

## Argon2id parameters

The wizard uses these parameters for the initial hash:

| Parameter     | Value   | Rationale                                                    |
| ------------- | ------- | ------------------------------------------------------------ |
| `time_cost`   | 3       | OWASP 2024 recommendation for Argon2id                       |
| `memory_cost` | 65536   | 64 MB — high enough to punish GPU attacks on commodity hw    |
| `parallelism` | 4       | Matches typical 4-core Postgres pods                         |
| `hash_len`    | 32      | 256-bit derived key; comfortably over collision bounds       |
| `salt_len`    | 16      | 128-bit salt; library default                                |

These are written into `backend/01_core/password.py` as module
constants so runtime verification (in `03_iam.01_auth`) uses the same
values. The hash output includes the parameters as part of the PHC
string format, so changing the parameters later does not break existing
hashes — the verifier reads the parameters from the stored hash.

## DB writes

Phase 3 performs exactly these writes, all in one transaction on an
admin Postgres connection:

```sql
BEGIN;

-- 1. The user row (fct_* — UUIDs and FKs only, no strings)
INSERT INTO "03_iam"."10_fct_users" (
    id,
    org_id,
    account_type_id,
    auth_type_id,
    is_active,
    is_test,
    created_by,
    updated_by
) VALUES (
    $user_id,                   -- UUID v7 generated by wizard
    $user_id,                   -- first admin is their own org (singleton org)
    :default_admin_id,          -- FK → 03_iam.06_dim_account_types.code='default_admin'
    :username_password_id,      -- FK → 03_iam.07_dim_auth_types.code='username_password'
    TRUE,
    FALSE,
    $user_id,                   -- self-created; reflexive FK
    $user_id
);

-- 2. EAV attributes — username, email, password_hash
INSERT INTO "03_iam"."20_dtl_attrs"
    (id, entity_type_id, entity_id, attr_def_id, key_text)
VALUES
    ($attr_id_1, :iam_user_entity_type, $user_id, :attr_username,      $username),
    ($attr_id_2, :iam_user_entity_type, $user_id, :attr_email,         $email),
    ($attr_id_3, :iam_user_entity_type, $user_id, :attr_password_hash, $argon2_hash);

-- 3. Final system_meta update — THIS is the write that marks "installed"
UPDATE "00_schema_migrations".system_meta
   SET installed_at           = CURRENT_TIMESTAMP,
       installer_version      = :installer_version,
       first_admin_username   = $username,
       first_admin_created_at = CURRENT_TIMESTAMP,
       updated_at             = CURRENT_TIMESTAMP
 WHERE id = 1;

COMMIT;
```

If any statement fails, the entire transaction aborts and `installed_at`
stays NULL. Re-running the wizard resumes at Phase 3 and re-prompts for
the admin credentials (the operator did not type a valid set the first
time, most likely).

## Why username + email + password_hash are EAV

Per the database conventions (`.claude/rules/common/database.md`):

- `fct_*` tables hold UUIDs and FKs only — no strings, no JSONB.
- All human-readable or PII-flavoured fields live in `dtl_attrs` (EAV).
- Every attribute is registered once in `dim_attr_defs` and then
  referenced by `attr_def_id` on every `dtl_attrs` row.

So `10_fct_users` has no `username` column. Instead the `03_iam`
bootstrap migration seeds three rows in `dim_attr_defs`:

| code            | entity_type | value_column | purpose                  |
| --------------- | ----------- | ------------ | ------------------------ |
| `username`      | iam_user    | key_text     | Login identifier         |
| `email`         | iam_user    | key_text     | Contact / future resets  |
| `password_hash` | iam_user    | key_text     | Argon2id PHC-format hash |

And `v_users` JOINs `10_fct_users` against `20_dtl_attrs` three times
to project the human shape. Phase 3 inserts into the raw tables; the
runtime reads `v_users`.

## In scope

- Username / email / password prompts with the validation rules above
- Argon2id hashing with the documented parameters
- Generating one UUID v7 for the user row
- Insert into `10_fct_users` (singleton reflexive-org pattern)
- Insert three rows into `20_dtl_attrs` for username / email /
  password_hash
- Update `system_meta` with `installed_at`, `installer_version`,
  `first_admin_username`, `first_admin_created_at`
- Running all four writes in one transaction via the admin Postgres
  connection
- Returning the new user's UUID to the wizard for the "next steps"
  banner

## Out of scope

- Email verification (future)
- Password strength meter beyond the minimum rules (future)
- MFA enrolment at install time (v1 IAM has no MFA at all)
- OAuth / SSO admin (v1 IAM has no OAuth at all)
- Multi-admin install — Phase 3 creates exactly one user; subsequent
  admins are created via the IAM HTTP API after first login
- Seeding any other rows in `03_iam` (audit, permissions, etc.) —
  IAM's own bootstrap migration handles non-user setup
- Password reset flow — if the first admin forgets the password, the
  operator deletes the row manually and re-runs Phase 3 via
  `tennetctl setup --resume`

## Acceptance criteria

### Prompts

- [ ] Username prompt rejects input with spaces, non-ASCII, or symbols
      outside `._-`
- [ ] Username prompt rejects input shorter than 3 or longer than 64
- [ ] Email prompt rejects anything that Pydantic `EmailStr` rejects
- [ ] Password prompt uses echo-suppressed input
- [ ] Password prompt rejects anything under 12 characters
- [ ] Password prompt rejects all-alpha passwords
- [ ] Confirmation prompt must match exactly
- [ ] Non-interactive mode with missing or invalid password exits
      non-zero before touching the DB

### Hash

- [ ] The hash stored in `password_hash` attribute is a valid
      Argon2id PHC string (`$argon2id$v=19$m=65536,t=3,p=4$...`)
- [ ] The hash verifies against the original password via the same
      `argon2-cffi` wrapper the runtime auth code uses
- [ ] Wrong password does not verify
- [ ] The parameters in the stored hash match the documented defaults

### DB writes

- [ ] Exactly one row inserted in `10_fct_users` with
      `account_type_id` resolved from `dim_account_types` WHERE
      `code = 'default_admin'`
- [ ] Exactly three rows inserted in `20_dtl_attrs` (username, email,
      password_hash)
- [ ] `created_by` and `updated_by` on the user row equal the user's
      own id (reflexive self-reference; the installer has no identity)
- [ ] `system_meta.installed_at IS NOT NULL` after Phase 3 commits
- [ ] `system_meta.first_admin_username` matches the entered username
- [ ] `system_meta.first_admin_created_at` matches the user row's
      `created_at` within the same second
- [ ] If any statement raises, the entire transaction aborts — nothing
      is written to either table

### Idempotency

- [ ] Running Phase 3 when `system_meta.installed_at IS NOT NULL`
      refuses to run with "Phase 3 already complete" (operator must
      clear the row manually to re-run); `detect_phase` normally skips
      straight to Phase 4 in this case
- [ ] Running Phase 3 after an earlier crash (no user row, no
      `installed_at`) re-prompts and creates a fresh user — no
      leftover state to clean up

## Dependencies

- Depends on: `03_iam.00_bootstrap` (needs `10_fct_users`,
  `06_dim_account_types`, `07_dim_auth_types`, `20_dtl_attrs`,
  `06_dim_entity_types`, and `07_dim_attr_defs` all seeded)
- Depends on: `01_core/password.py` (the Argon2id wrapper that
  `03_iam.01_auth` also uses — Phase 3 must produce hashes that
  runtime verify will accept)
- Depends on: `01_core/id.py` (UUID v7 generator used for the user id)
- Depended on by: `00_wizard` orchestration, and implicitly by every
  runtime feature via the `installed_at IS NOT NULL` boot check
