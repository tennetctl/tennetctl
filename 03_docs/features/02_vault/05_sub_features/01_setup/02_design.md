# T-Vault Setup — Design

## The pluggable UnsealBackend abstraction

Every deployment picks one unseal mode at install time. The selected mode is
written to `10_fct_vault.unseal_mode_id` and is read on every process boot to
instantiate the right backend class.

```python
# backend/02_features/vault/setup/unseal_backend.py

class UnsealBackend(Protocol):
    """Contract every unseal mode must satisfy."""
    code: str  # "manual" | "kms_azure" | "kms_aws" | "kms_gcp"

    async def init(self, config: dict) -> InitResult:
        """Run exactly once during install. Generate MDK, wrap/encrypt it,
        return both the plaintext MDK (held in memory for the rest of install)
        and the persistence payload that will be written to 10_fct_vault."""

    async def unseal(self, vault_row: dict, user_input: str | None) -> bytes:
        """Return the plaintext MDK. Called on every process boot after the
        first. `user_input` is the Root Unseal Key for manual mode, None for
        KMS modes (which auto-fetch via workload identity)."""

    def auto_unseal(self) -> bool:
        """True if this backend can unseal with no user interaction."""
```

### Backend registry

```python
UNSEAL_BACKENDS: dict[str, type[UnsealBackend]] = {
    "manual":    ManualUnsealBackend,
    "kms_azure": AzureKeyVaultUnsealBackend,     # from 02_kms_azure
    "kms_aws":   AwsKmsUnsealBackend,            # from 03_kms_aws (PLANNED)
    "kms_gcp":   GcpKmsUnsealBackend,            # from 04_kms_gcp (PLANNED)
}

def get_backend(mode_code: str) -> UnsealBackend:
    cls = UNSEAL_BACKENDS.get(mode_code)
    if cls is None:
        raise ValueError(f"Unknown unseal mode: {mode_code}")
    return cls()
```

Each KMS sub-feature registers its class by importing from this module.
`01_setup` only ships `ManualUnsealBackend`; the others plug in via their own
sub-feature code.

## ManualUnsealBackend (this sub-feature's only concrete implementation)

### Key inputs

Two 64-character lowercase hex strings (SHA-256 outputs), generated by the
operator:

```text
Root Unseal Key  —  64 hex chars  —  used to encrypt/decrypt the MDK
Root Read Key    —  64 hex chars  —  reserved for future root-read operations;
                                      stored only as a BLAKE2b hash
```

Neither key is stored in any form that allows recovery. They are:

- Verified on use via BLAKE2b hash comparison
- Discarded from memory after use

### Cryptographic operations

```text
init(config={unseal_key, read_key}):
  mdk              = os.urandom(32)
  nonce            = os.urandom(12)
  mdk_ciphertext   = AES256GCM(key=unseal_key).encrypt(nonce, mdk, aad=None)
  unseal_key_hash  = BLAKE2b(unseal_key)
  read_key_hash    = BLAKE2b(read_key)
  → persist { mdk_ciphertext, mdk_nonce=nonce, unseal_key_hash, read_key_hash,
              unseal_mode_id = manual, status_id = sealed }
  → return mdk (held in memory for the rest of install)

unseal(vault_row, user_input=unseal_key):
  provided_hash = BLAKE2b(user_input)
  IF provided_hash != vault_row.unseal_key_hash → raise WrongUnsealKey
  mdk = AES256GCM(key=user_input).decrypt(vault_row.mdk_nonce,
                                          vault_row.mdk_ciphertext)
  return mdk

auto_unseal():
  return False
```

## Startup state machine

```text
process start
    │
    ▼
read $DATABASE_URL → bootstrap DSN
    │
    ▼
connect to Postgres as write role
    │
    ▼
SELECT * FROM v_vault
    │
    ▼
IF unseal_mode == 'manual':
    start HTTP in SEALED mode
    only /v1/vault/unseal, /v1/vault/status, /v1/vault/seal, /healthz respond
    ALL OTHER routes → 503 VAULT_SEALED
    wait for operator POST /v1/vault/unseal
    on success → load write_dsn/read_dsn/other secrets from vault
    → flip to OPERATIONAL

IF unseal_mode IN ('kms_azure', 'kms_aws', 'kms_gcp'):
    backend = get_backend(unseal_mode)
    mdk = await backend.unseal(vault_row, user_input=None)
    (calls Azure KV / AWS KMS / GCP KMS using pod's workload identity)
    on success → load write_dsn/read_dsn/other secrets from vault
    → start HTTP directly in OPERATIONAL mode
    on failure → crash the pod; K8s restart loop surfaces the problem loudly
```

The critical property: in KMS modes, the pod becomes operational with zero
human interaction. HPA can scale up a hundred pods, and each one auto-unseals
independently via its own workload identity token exchange.

## Data model

### `01_dim_vault_statuses`

| id  | code     | label    |
| --- | -------- | -------- |
| 1   | sealed   | Sealed   |
| 2   | unsealed | Unsealed |

### `02_dim_unseal_modes`

| id  | code      | label                 |
| --- | --------- | --------------------- |
| 1   | manual    | Manual (operator key) |
| 2   | kms_azure | Azure Key Vault       |
| 3   | kms_aws   | AWS KMS               |
| 4   | kms_gcp   | GCP KMS               |

### `10_fct_vault`

| Column          | Type        | Notes                                                |
| --------------- | ----------- | ---------------------------------------------------- |
| id              | VARCHAR(36) | UUID v7 PK                                           |
| status_id       | SMALLINT    | FK → 01_dim_vault_statuses                           |
| unseal_mode_id  | SMALLINT    | FK → 02_dim_unseal_modes. Immutable after init.      |
| unseal_config   | TEXT        | JSONB-as-text. KMS-backend metadata. NULL in manual. |
| wrapped_mdk     | TEXT        | MDK wrapped by cloud KMS. NULL in manual.            |
| mdk_ciphertext  | TEXT        | AES-256-GCM MDK ciphertext. NULL in KMS modes.       |
| mdk_nonce       | TEXT        | GCM nonce for mdk_ciphertext. NULL in KMS modes.     |
| unseal_key_hash | TEXT        | BLAKE2b of Root Unseal Key. NULL in KMS modes.       |
| read_key_hash   | TEXT        | BLAKE2b of Root Read Key. NULL in KMS modes.         |
| initialized_at  | TIMESTAMP   | Set once on init. NULL = not yet initialized.        |
| created_at      | TIMESTAMP   | Row creation time.                                   |
| updated_at      | TIMESTAMP   | Updated on each seal/unseal.                         |

The per-mode CHECK constraint enforces that exactly one shape of key material
is present after init.

### `v_vault` view

```sql
SELECT
    v.id,
    s.code  AS status,
    m.code  AS unseal_mode,
    v.initialized_at,
    v.updated_at
FROM "02_vault"."10_fct_vault" v
JOIN "02_vault"."01_dim_vault_statuses" s ON v.status_id = s.id
JOIN "02_vault"."02_dim_unseal_modes"  m ON v.unseal_mode_id = m.id;
```

All key-material columns are excluded from the view — they are never surfaced
via any API response.

## Service layer

```text
vault_setup_service.py

init_vault(conn, mode: str, config: dict) -> dict
  - Check no existing row in 10_fct_vault (idempotency guard)
  - backend = get_backend(mode)
  - init_result = await backend.init(config)
  - INSERT into 10_fct_vault with status_id=sealed, unseal_mode_id=mode,
    plus the per-mode key-material columns from init_result
  - Emit vault.initialized event
  - Return { status: "sealed", unseal_mode: mode, initialized_at: ... }

unseal_vault(conn, vault_state: VaultState, user_input: str | None) -> dict
  - Load row from v_vault (status, unseal_mode)
  - IF status == unsealed → 409 VAULT_ALREADY_UNSEALED
  - Load full row from 10_fct_vault
  - backend = get_backend(row.unseal_mode_code)
  - IF NOT backend.auto_unseal() AND user_input IS NULL → 400 UNSEAL_KEY_REQUIRED
  - IF backend.auto_unseal() AND user_input IS NOT NULL → 409 AUTO_UNSEAL_ACTIVE
  - mdk = await backend.unseal(row, user_input)
  - vault_state.mdk = mdk
  - UPDATE 10_fct_vault SET status_id = unsealed, updated_at = now()
  - Emit vault.unsealed event
  - Return { status: "unsealed", unseal_mode: row.unseal_mode_code }

seal_vault(conn, vault_state: VaultState) -> None
  - Clear vault_state.mdk
  - UPDATE 10_fct_vault SET status_id = sealed, updated_at = now()
  - Emit vault.sealed event

get_vault_status(conn) -> dict
  - SELECT from v_vault
  - backend = get_backend(row.unseal_mode_code)
  - Return { status, unseal_mode, auto_unseal: backend.auto_unseal(),
             initialized_at }
```

## VaultState singleton

An in-memory singleton injected as a FastAPI dependency. Holds the decrypted
MDK while the vault is unsealed. Cleared on seal or process restart.

```python
class VaultState:
    mdk: bytes | None = None

    @property
    def is_unsealed(self) -> bool:
        return self.mdk is not None
```

All secret encrypt/decrypt operations receive `vault_state.mdk` directly.
If `vault_state.mdk is None`, they raise `503 VAULT_SEALED`.

## Startup hook

```python
# backend/01_core/startup.py

async def startup_unseal(vault_state: VaultState, db_pool):
    """Called once during FastAPI startup."""
    async with db_pool.acquire() as conn:
        row = await conn.fetchrow("SELECT * FROM \"02_vault\".v_vault")
        if row is None:
            raise RuntimeError("Vault not initialized — run tennetctl setup")

        backend = get_backend(row["unseal_mode"])
        if backend.auto_unseal():
            # KMS modes: unseal automatically using workload identity
            full_row = await conn.fetchrow(
                "SELECT * FROM \"02_vault\".\"10_fct_vault\" LIMIT 1"
            )
            vault_state.mdk = await backend.unseal(dict(full_row), None)
            logger.info("Vault auto-unsealed via %s", row["unseal_mode"])
        else:
            # manual mode: wait for operator POST /v1/vault/unseal
            logger.info("Vault in sealed mode — waiting for operator unseal")
```

## API layer

| Method | Path               | Description                                      |
| ------ | ------------------ | ------------------------------------------------ |
| POST   | `/v1/vault/unseal` | Unseal (manual mode only; 409 in KMS modes)      |
| POST   | `/v1/vault/seal`   | Seal the vault (clears MDK from memory)          |
| GET    | `/v1/vault/status` | Returns status + unseal mode + auto_unseal flag  |

Note: `POST /v1/vault/init` is NOT exposed as an HTTP route in v1. Init is
only callable from the install wizard's service-layer integration.

## Security considerations

- The init payload is the most sensitive data in the system. In manual mode
  the two root keys never leave process memory. In KMS modes the install
  wizard only receives the KMS key reference (key vault URL, KMS ARN) — never
  actual cryptographic material.
- `mdk_ciphertext`, `mdk_nonce`, `unseal_key_hash`, `read_key_hash`,
  `wrapped_mdk`, and `unseal_config` are **never included in any API response**
  — only in the database.
- All endpoints enforce HTTPS in production (enforced at the reverse proxy).
- Auto-unseal trade-off: in KMS modes, an attacker who compromises the pod's
  runtime (RCE) gets the MDK in memory and can decrypt everything until the
  pod dies. Manual mode has the same property once unsealed, but the blast
  radius is smaller (only while the pod is running, not across restarts).
  This is an accepted trade-off for K8s-native operation.
